# Digest — 2026-06-04

*Distilled from a twice-daily AI-engineering sweep. Backdated.*

## In plain English

The plumbing under AI assistants quietly became a real industry today. A protocol called MCP (Model Context Protocol — think USB for AI tools) hit 110 million monthly downloads in roughly half the time it took React, and GitHub's registry of these connectors crossed nearly 2,000 entries with 407% growth since September. The clearest voices in the field are now telling builders to stop reaching for "agents" — software that figures out its own steps — for every problem; if you can draw the decision tree on a whiteboard, you should build a fixed workflow instead. The honest summary: the smart-AI race has turned into a plumbing race, and the people winning it are the ones picking smaller, simpler shapes.

## TL;DR

- **Most important:** MCP has graduated from demo protocol to enterprise infrastructure — 110M monthly downloads, donated to the Linux Foundation, GitHub's registry at ~2,000 servers and 407% growth.
- **Most overhyped:** "Build an agent" — Barry Zhang's pushback is that most agent projects should have been workflows; the model owning the plumbing is a cost, not a feature.
- **Emerging:** Skills-over-MCP, progressive discovery, and programmatic tool calling — the shape of how a model talks to the world is changing under the hood.

## Today's reading

### 01 — Don't build agents — build skills
*Barry Zhang & Mahesh Murag · Anthropic* · [source](https://www.youtube.com/watch?v=CEvIs9y1uog)

Barry Zhang from Anthropic argues that most "AI agent" projects shouldn't be agents at all. If you can already draw the decision tree on a whiteboard, the right thing to build is a fixed workflow where you wire up each step yourself. Agents — software that decides its own next move — are for genuinely ambiguous tasks, and they cost more accuracy, control, and money than people admit. The headline rule: don't reach for the autonomous version of a problem you've already solved.

Zhang's framing: *"With a workflow, you own the plumbing. With an agent, the model owns the plumbing."* Agents are defined as *"models using tools in a loop"* dynamically directing their own processes, anchored by three things — the environment, the tools, and the system prompt. His first rule is **don't build agents for everything**: the criterion is ambiguity. If the decision tree is pre-mappable, build a workflow and optimise each node. Skills over agents whenever the task is well-understood.

### 02 — MCP hits 110M monthly downloads
*David Soria Parra · Anthropic* · [source](https://www.youtube.com/watch?v=v3Fr2JR47KA)

David Soria Parra co-created MCP — the wiring standard that lets AI assistants plug into outside tools — and he laid out where it's going next. The big number: 110 million downloads a month, hit in about half the time React took to reach the same level. It's now used by OpenAI, Google, and thousands of frameworks, and Anthropic has handed it over to the Linux Foundation so no single company owns it. The takeaway for non-builders: the connector layer for AI just stopped being an Anthropic project and started being public infrastructure.

Two client-side patterns are the story for 2026. **Progressive discovery** — lazy-loading tools on demand rather than dumping every tool spec into context up front — solves context bloat at enterprise scale. **Programmatic tool calling** — having the model write scripts that compose multiple tool calls — *"turns the agent from a tool-caller into a programmer."* Roadmap: stateless transport, async primitives for agent-to-agent comms, SDK v2 in TypeScript/Python, cross-app SSO, well-known-URL server discovery, and skills-over-MCP for shipping domain knowledge alongside servers.

### 03 — Stop converting REST APIs to MCP
*Jeremiah Lowin · AI Engineer Code Summit 2025* · [source](https://www.youtube.com/watch?v=UQjAZrTM0MI)

Jeremiah Lowin, who built the popular FastMCP toolkit, says most of today's MCP servers are bad products. The reason: people auto-generate them from web APIs designed for humans clicking around — hundreds of tiny endpoints, infinitely flexible. That's fine for code, but for an AI model every single call is expensive, and chaining ten little ones to do one job is slow, error-prone, and burns money. The fix is human work: pick the few combined actions an assistant actually needs, and throw the rest away.

Three failure modes: **context cost** (LLMs process every tool name + description + parameter on each reasoning cycle), **atomicity mismatch** (REST is generous because iteration is cheap for code; for LLMs iteration is *"slow, error-prone, and burns through tokens"*), and **context pollution** — *"your agent stops being a helpful assistant and becomes an obsessive API librarian."* Lowin's prescription: *"Stop converting your REST APIs. Start curating them."* Bootstrap with `FastMCP.from_openapi()`, then ruthlessly curate. Start from the agent story: *"As an agent, given {context}, I use {tools} to achieve {outcome}."*

### 04 — GitHub registry hits 2,000 servers
*GitHub team · AI Engineer Paris 2025* · [source](https://www.youtube.com/watch?v=661yQKSoKP8)

GitHub showed the numbers behind MCP going mainstream inside big companies. Their public registry of connectors now lists nearly 2,000 entries — up 407% since September. Pinterest's internal deployment alone logged 66,000 tool calls in a month from 844 employees. Slack, Salesforce, Stripe, HubSpot, Shopify, Notion, Linear, Figma, and Cloudflare all run official versions. This is the moment a protocol stops being a demo and starts being IT infrastructure.

The unglamorous work is now the work: transport scalability, agent-to-agent comms protocols, governance (who can access which tools), and enterprise readiness — SSO, audit logging, rate limiting. The gap between *"works in Claude Desktop"* and *"works for 844 users"* is where the next year of engineering lives. Read alongside Lowin: curation + Lowin's three failure modes are what stand between a registry entry and a production deployment.

### 05 — MCP as the multi-agent connector layer
*Dan Mason · AI Engineer Summit NYC 2025* · [source](https://www.youtube.com/watch?v=4CB8zDdShzA)

Dan Mason's workshop reframed MCP from "the thing that lets one AI use one tool" to "the wiring between teams of AIs and the real world." Each AI in a multi-agent setup gets its own scoped slice of tools through MCP, with a router agent deciding who handles what. The deeper point is for product people: instead of writing code, you describe roles, the tools each role can touch, and the outcome you want — and let the system figure out the rest.

Built on LangGraph (compatible with AutoGen, CrewAI). Mason's framing: *"Rethink how to express business logic as a set of roles, tools, and desired outcomes"* rather than as code. Each agent gets scoped tool access through MCP; the orchestrator routes between agents. Reads as a useful counterweight to Zhang — multi-agent has a real shape, but the *workflow vs agent* question still applies at every node inside the system.

## Skip the hype

- **"Just build an agent"** — Zhang's gauntlet: if you can pre-map the decision tree, a workflow beats an agent on accuracy, control, and cost. Most agent projects are smuggled workflows wearing a costume.
- **Auto-converted REST→MCP servers** — Lowin's verdict is blunt: bad products. A registry entry that ships every endpoint isn't a tool, it's a context-pollution machine; curation is the actual engineering.

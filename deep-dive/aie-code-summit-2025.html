Deep dive · AI Engineer Code Summit 2025

Seven talks from the year's most engineering-honest AI conference.

AI Engineer Code Summit · Nov 19-22, 2025 · NYC · channel ↗

If you wanted to know what people actually doing the work — not selling the keynote — were saying about AI coding agents in late 2025, this conference is the answer. The thread across every talk: simplification beats scaffolding. Strong models with the right harness beat baroque pipelines. Verification is the work. We pulled the seven non-marketing talks, paired each with the speaker's written companion, and decoded them.

01
practitioner Dex Horthy · HumanLayer

Research, Plan, Implement — discipline beats throwing tickets at an agent

“The dumb zone” — Horthy's term for context windows past ~40% utilization, where model quality degrades sharply.

Horthy argues that AI coding tools fail in real codebases not because the models are weak but because we let them try too much at once. His fix is almost boring: make the agent research the code first, write down a plan, get human alignment, then implement — and aggressively clear context between phases. The unsexy procedural discipline is the whole point. He's since published "Everything We Got Wrong About RPI" admitting the v1 framework had three failure modes at scale.

RPI is a context-engineering workflow: stage 1 forbids implementation, forces fact-gathering and produces a research artifact; stage 2 produces an architectural plan reviewed before any code; stage 3 implements against the locked plan. Horthy advocates intentional compaction — explicitly summarising and resetting between phases rather than relying on auto-summarization. The v2 framework (QRSPI) splits these into eight stages and hides the ticket during research to prevent premature opinion-formation.

02
practitioner Nik Pash · Cline

Stop scaffolding. Strong models with simple tools beat RAG + multi-agent

“The hardest questions and queries in a codebase require several hops. Vanilla retrieval only works for one hop.” — Aman Sanger (Cursor), quoted approvingly.

For two years everyone added vector databases, RAG pipelines, and multi-agent orchestrators to coding agents. Pash's argument is that as the underlying models got better, these scaffolds started hurting more than helping — the agent reasons better when you just give it the tools a human engineer uses (open file, search, inspect imports) and let it explore. It's a the application layer is shrinking thesis specifically applied to code.

Embeddings flatten the logical structure of code into similarity scores, which loses the hop-structure of real questions (call sites, type inheritance, dependency graphs). Pash recommends agentic tool-use (filesystem + AST + grep) over retrieval pipelines, claiming context-quality not context-size is now the bottleneck. ClineBench is positioned as the eval to standardise this claim. Pairs directly with the multi-agent debunking from Stanford and Salesforce — same architectural simplification thesis.

03
primary Will Brown · Prime Intellect

RL environments are the new bottleneck — and they should be open

“If high-quality environments remain expensive and closed, open-source models will fall further behind.”

Pretraining data scaled because the web was free. The next phase of model progress depends on reinforcement learning, and RL needs environments — little simulated worlds where models can try things, fail, and learn. Right now those environments live inside the big labs and are the secret sauce. Brown's pitch is to do for environments what Hugging Face did for models: a public hub where anyone can publish, share, and run them.

Environments are distributed as Python packages (pyproject.toml, wheels) in a proper registry, with sandboxed Verifier Environments for safe code execution. Native integration with Prime Intellect's open-source prime-rl trainer. The architectural pitch frames environments as the unit of post-training capability — not models or datasets. 30+ contributors in private beta, 2,500+ environments referenced via NVIDIA Nemotron Coalition partnership.

04
primary Joel Becker · METR

The RCT result: experienced devs were 19% slower with AI — and 20% sure they were faster

“When developers are allowed to use AI tools, they take 19% longer to complete issues — a significant slowdown that goes against developer beliefs and expert forecasts.”

Becker's group at METR ran an actual randomised controlled trial — rare in this space — and got a deeply counterintuitive result. AI made experienced devs slower on their own codebases, even though those same devs thought it sped them up. The talk is the bridge between his benchmarks-vs-productivity research and the AI Engineer audience's daily experience. It's the strongest evidence-based check on benchmarks went up so productivity must be up.

16 developers, 246 tasks, +19% completion time with AI (95% CI: −26% to +9%); roughly 50% of test-passing SWE-bench Verified PRs would not be accepted by maintainers — implying benchmark scores are upper bounds, not productivity estimates. The perception gap = 39 absolute points (devs felt 20% faster, were 19% slower). METR has since (Feb 2026) redesigned the study for late-2025 tools; Becker is the right voice to track this longitudinally.

05
vendor Eno Reyes · Factory AI

If your codebase doesn't tell the agent it's wrong, the agent will be wrong

“What is AI? A great pattern recognizer. So great code in the pattern means great code out.”

Reyes argues most teams blame the model when agents fail in their codebase, but the real problem is that their codebase doesn't tell the agent when it's wrong. If your linter is permissive, your tests are sparse, your docs are stale, and your style is inconsistent, the agent has no feedback loop. Make the codebase a strict teacher and agents start working. Useful frame — but the headline 5-10× return number is a vendor claim with no published methodology.

Concrete recommendations: opinionated linters, automated formatting, high-coverage tests, machine-readable docs, and verification loops baked into CI. The pitch dovetails with the broader evals-as-product thesis (Pash on ClineBench, METR on benchmark/productivity gap). Treat the 5-10× number as marketing; the underlying technical claim about verification loops is solid and overlaps with how serious shops are structuring their agent stacks.

06
primary Jacob Kahn · FAIR (Meta)

Train code models on execution traces, not just code tokens

“[CWM is] mid-trained on observation-action trajectories from Python interpreter and agentic Docker environments.”

Almost all code models today learn from code-as-text — they see the words a program is made of, not what the program does when it runs. Kahn's group at Meta trained CWM on traces of programs executing: variables changing, files getting written, processes running. The model picks up an implicit feel for computation, not just syntax. They open-sourced the weights at 32B parameters, which is generous by frontier-lab standards.

CWM uses observation-action trajectories from Python interpreters and agentic Docker environments during mid-training, then multi-task RL on verifiable coding/math/multi-turn SWE tasks. Pass@1: 65.8% SWE-bench Verified (with test-time scaling), 68.6% LiveCodeBench, 96.6% Math-500, 76.0% AIME 2024. Competitive with much larger closed models at 32B if reproducible. Relevant to anyone debugging why pure-pretrained-LLM agents hallucinate non-existent function signatures or misunderstand state mutation.

07
vendor Tobin South · WorkOS (also Anthropic MCP spec contributor)

MCP demos pass; MCP production fails — the missing layer is auth, scoping, audit

“MCP has emerged as the standard interface for AI agents, supported by major platforms like Cursor, Microsoft Copilot, ChatGPT, and Claude.”

There are now thousands of MCP servers but almost all of them assume one user, one machine, full trust. The moment you try to put one inside a company with a security team, none of those assumptions hold. South's talk is the practical checklist — auth flow, scoping, who can connect what to which data — for crossing from demo to production. Pairs cleanly with Jeremiah Lowin's earlier Your MCP Server is Bad — they are complementary critiques.

Concrete enterprise requirements: OAuth-based authorization with client ID metadata (so orgs can allow-list which AI apps connect), SSO/SAML for human provisioning, scoped capabilities per client, audit logging. References FastMCP 2.0 + AuthKit as a reference stack. Note: South wears two hats (Anthropic MCP spec work + WorkOS commercial) — the spec-level claims are higher-trust than the WorkOS-specific recommendations.
